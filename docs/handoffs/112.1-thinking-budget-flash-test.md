# 112.1 — thinkingBudget Flash test for LEAD prose compliance

## What this is

The carry from HO 112: morphological variants of banned phrases ("notably" when "notable" is banned, "significantly" after explicit ban) leaked into the LEAD under Gemini 2.5 Flash's default thinking budget. HO 112 already shipped the structural prose work. This handoff is the targeted retry with raised thinkingBudget, scoped as the cheap empirical test before any model bump.

Two-phase. Phase 1 is diagnostic and ends with a HALT for sign-off. Phase 2 commits or reverts based on the Phase 1 result.

No commits in Phase 1.

## Cost context

Pricing pulled from ai.google.dev/gemini-api/docs/pricing on 2026-05-19. Bumping thinkingBudget on Gemini 2.5 Flash raises output tokens (thinking tokens are billed at the output rate). Estimated annual cost change at this report cadence: $0.26 → ~$0.65. Not the decision driver. Full reasoning lives in the LLM model selection memo.

## Phase 1 — diagnostic

### Step 1. Locate the Flash call

Weekly report generation lives in `lib/report-generation.ts`. Find the `gemini-2.5-flash` call (via `@google/genai`). Note whether `thinkingConfig` is already set anywhere in the project.

### Step 2. Verify SDK syntax for thinkingBudget

Before changing anything, confirm the correct `thinkingConfig.thinkingBudget` syntax for `@google/genai` against current docs at https://ai.google.dev/gemini-api/docs/thinking. Do not rely on training assumptions. Valid range for 2.5 Flash is 0–24576 tokens; default is dynamic.

### Step 3. Add thinkingBudget = 8192

Set `thinkingConfig: { thinkingBudget: 8192 }` on the Flash call. One change, one location. Don't touch the SYSTEM_PROMPT, the buildUserPrompt enrichment, or the response parsing — HO 112 shipped those and they're out of scope.

Rationale for 8192: substantial bump from default but well under the 24576 ceiling. Enough headroom for the model to self-check the output against the banned list. If 8192 still leaks variants, raising further is unlikely to help and Pro is the next step.

### Step 4. Regenerate 3 historical weeks

Regenerate the LEAD for the same three weeks HO 112 used as verification:

- 2026-05-18 (prediction markets ethics throughline)
- 2026-05-11 (HR 8469 floor→committee setback)
- 2026-04-27 (enactments week)

Output to a scratchpad file at `tmp/leads-thinking-budget-test.md`. Do NOT write to the reports table or overwrite existing report records. Format: for each week, show original LEAD on top, regenerated LEAD on bottom, with the week's date as a header.

### Step 5. Banned-phrase scan

Scan each new LEAD against the banned-phrase list from the existing SYSTEM_PROMPT. Include morphological variants. At minimum check stem matches for: notable/notably, significant/significantly, comprehensive, robust, leverage, underscore, delve, dive, navigate (figurative), bolster, foster, harness, pivotal, transformative, groundbreaking, seamless, intricate, multifaceted, holistic, testament, landscape (figurative), realm.

For each hit, log the phrase and the surrounding sentence.

### Step 6. Report in chat and HALT

Post to chat:

1. The three side-by-side LEADs (or a link/path to the scratchpad).
2. Banned-phrase scan results per week (clean or itemized hits).
3. Subjective read on whether each new LEAD count-leads (opens with a statistic) or synthesizes a throughline.
4. The per-call token usage from the SDK's `usageMetadata` (input, output, thinking tokens broken out if reported separately).

HALT. Do not proceed to Phase 2 without sign-off.

## Phase 2 — only after Phase 1 sign-off

Two paths, selected based on the Phase 1 outcome.

### Path A — bump cleared the bar

Commit the thinkingConfig change. Commit message: `feat(reports): raise Flash thinkingBudget to 8192 for LEAD compliance (HO 112.1)`.

Update `SKILL.md` under "Things to watch for" with a one-line note: "Weekly report LEAD generation uses Gemini 2.5 Flash with thinkingBudget=8192, raised in HO 112.1 to address morphological variant leakage of banned phrases under the default budget."

Working tree clean, push.

### Path B — bump did not clear the bar

Revert the config change. Don't ship a half-fix. Report back the specific failure modes (which variants leaked, count-leading rate) so the next handoff — likely Pro escalation, possibly a regenerate-on-violation layer — can be scoped from real data, not speculation.

## Out of scope

- Adding regenerate-on-violation logic. Separate handoff after the model choice is settled.
- Touching SYSTEM_PROMPT or buildUserPrompt. HO 112 closed that work.
- Generating reports for weeks beyond the three verification weeks.
- Bumping to Pro. If Phase 1 fails, that's the next chat conversation, not the next handoff in this session.

**Phase 1:** Three regenerated LEADs in `tmp/leads-thinking-budget-test.md`, scan results and subjective read posted to chat, no commits.

**Phase 2:** Either the thinkingConfig bump committed and SKILL.md updated (Path A), or the bump reverted with clean working tree (Path B).
