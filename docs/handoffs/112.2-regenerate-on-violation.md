# 112.2 — regenerate-on-violation for banned-phrase compliance

## What this is

HO 112.1 shipped thinkingBudget=8192 which fixed count-leading (3/3 LEADs now synthesize) but banned-phrase morphological variants still leak at ~1-in-3 frequency. The previous handoff identified regenerate-on-violation as the deterministic fix; this implements it.

Phased. Phase 1 implements the retry logic and verifies on the known failure case (2026-05-11). Phase 2 commits or reverts based on Phase 1.

No commits in Phase 1.

## Design

### The check

Extract the banned-phrase list from the existing SYSTEM_PROMPT in `lib/report-generation.ts`. Represent each single-word entry as a stem that matches with any suffix. Store as a TypeScript constant at module top:

```typescript
const BANNED_STEMS: string[] = [/* extracted from SYSTEM_PROMPT */];
```

Derive the regex array on module load:

```typescript
const BANNED_REGEX = BANNED_STEMS.map(stem =>
  new RegExp(`\\b${stem}\\w*\\b`, 'i')
);
```

This matches `significant`, `significantly`, `significants` (suffix variants) without matching `insignificant` (prefix-modified, breaks the `\b` boundary).

Multi-word phrases ("in today's fast-paced world", "it's not just X but Y") are out of scope for v1. Single-word morphological variants only.

### Single source of truth

After this handoff, the SYSTEM_PROMPT must interpolate from `BANNED_STEMS` rather than inlining the list as prose. If the prompt and the regex array drift, the retry will fire on words the model wasn't told to avoid. Pick one source, derive the rest.

### The retry

Wrap the existing Flash call. After the response comes back, scan the full generated text (not just the LEAD; leakage anywhere is leakage). If any regex matches, retry once with a corrective instruction appended to the prompt:

```
Your previous response used the banned phrase "{found_phrase}". Regenerate the report without using that phrase or any morphological variant of it.
```

Use the *specific* phrase that matched, not a generic "you violated a rule." Models are better at avoiding named negatives.

Cap at 1 retry. If the second attempt also leaks, log the violation and ship the second output. Don't loop further; the Vercel 60s cron ceiling matters and a model that fails twice will likely fail a third time.

Keep `thinkingConfig: { thinkingBudget: 8192 }` on the retry call too. The retry needs reasoning budget to actually self-correct.

### Logging

`console.log` is enough; don't add a new DB table for this. Format:

```
[REPORT-RETRY] week=2026-05-11 attempt=1 violations=["significantly"] cleaned=true
```

If 112.2 ships and we want telemetry later, Vercel log search will surface these. A `cron_runs`-style table can be added in a future handoff if the retry rate becomes interesting.

## Phase 1 — diagnostic

1. Implement `BANNED_STEMS` extraction, regex array, and a `generateReportWithRetry` wrapper in `lib/report-generation.ts`. Don't refactor the surrounding code; minimal surface area.
2. Unify the SYSTEM_PROMPT to interpolate from `BANNED_STEMS` (so the prompt and the regex stay in sync).
3. Regenerate the 3 historical weeks (2026-05-18, 2026-05-11, 2026-04-27) to a scratchpad file at `tmp/leads-regenerate-test.md`. Use the same incomplete-week opt-out pattern from 112.1 for 05-18 (read-only scratchpad, no DB write).
4. For each week, capture:
   - Retry triggered? (yes/no)
   - If yes: violating phrase, original output snippet showing the violation, retry output, final scan result
   - If no: scan result (should be clean)
   - Token usage for original + any retry calls
5. Post results to chat. HALT.

**Expected result:** 2026-05-11 triggers one retry on "significantly" and the retry output is clean. 2026-05-18 and 2026-04-27 do not trigger retries (those LEADs were clean in 112.1's test).

If the actual result diverges materially from the expected, flag the divergence in the chat report and pause for direction before Phase 2.

## Out of scope

- Multi-word banned-phrase patterns. Single-word morphological variants only.
- DB-backed retry telemetry. `console.log` is the v1 telemetry.
- Touching SYSTEM_PROMPT prose content beyond the banned-list interpolation. HO 112 closed prose work.
- Scoping the scan to the LEAD specifically. Scan the full report output.
- Pro escalation. Separate handoff if 112.2 doesn't clear the bar.
