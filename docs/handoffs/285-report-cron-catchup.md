# HO 285 — Report cron: daily catch-up regenerator

The fix from the 284 probe. The weekly report drops because it's a single-shot Monday generation with no pipeline retry and no catch-up, so any transient hiccup loses the whole week. The probe found three such modes, each of which has dropped a report: a Turso cold-stall (#287, the boundedFetch 10+10), a Gemini 503 (#74/#82), and a slow-Gemini >55s (#37/#38). This adds a daily catch-up that regenerates a missing recent report, turning "one shot Monday" into "retried daily until it lands." It defeats all three at once and backfills the current 06-08 gap on its own.

Keep the Monday weekly cron as-is for timeliness. This is the safety net under it.

## The catch-up

Hook into the daily cron route (the daily sync; the probe pointed at `/api/sync`, confirm it and add this after the normal sync work).

Each daily run:
- Look at the last few completed weeks (a bounded window, e.g. the last 3-4; not all history). For each, check whether its report row exists in the reports table.
- If any are missing, regenerate exactly ONE per run, preferring the most recent missing week so the READ FULL target is restored first. One generation per run (~15-29s) keeps a single daily run well under budget. Multi-week gaps fill over consecutive days.
- If all present, do nothing.
- Idempotent: only generate when the row is missing; never overwrite an existing report.

Reuse the existing `generateWeeklyReport` / `gatherReportData` path for the target week (compute that week's date bounds). Don't fork the gen logic; just drive it for a missing week.

## Budget / config

- Confirm the daily route's maxDuration accommodates a normal gen (~15-29s) on top of its existing sync work. The weekly-report route is pinned at 60; match or raise on the daily route as needed, and confirm Hobby allows it.
- One gen per run keeps the worst case bounded (sync + one ~29s gen), so it shouldn't approach the cap.

## Not in this handoff (by design)

- The mode-specific patches (tightening the 503 retry ladder, warming/guarding the cold Turso connection on the weekly tick) are secondary; the catch-up makes them unnecessary for reliability. Leave them a backlog WATCH.
- #287's cold-stall is the HO 238 class, and one of `gatherReportData`'s ~10 queries may be a mis-planning aggregate of the same family as the 280 scout items. Don't chase it here; the catch-up lets the cron absorb the occasional stall. It folds into the systemic-pass WATCH.
