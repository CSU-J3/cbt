# 101 — 118th vs 119th laws-enacted comparison chart

## What this is

Ships the last roadmap theme-5 chart per the HO 99 audit. Cumulative laws enacted over the course of a Congress, overlaid for the 118th and 119th, on `/reports`. First chart on the reports surface — picks the visual idiom for that page, same way HO 81 picked it for the dashboard.

Honest answer to the framing question: how does this Congress's productivity compare to the last one at the same point in the term?

## Pre-flight (mandatory before any code)

The 118th Congress is not in the `bills` table. Per SKILL.md, CBT scopes the sync to the current Congress via `getCurrentCongress()`; 118th bills were never ingested. So this handoff needs a separate data source.

Two checks before scoping the backfill:

1. **Verify the Congress.gov `/law/{congress}` endpoint shape.** Fetch `https://api.congress.gov/v3/law/118?api_key=$CONGRESS_API_KEY&limit=10` and confirm: (a) the endpoint returns laws specifically (not bills); (b) the response includes enacted date and source bill ID per law; (c) total count via `pagination.count` is reasonable (118th expected ~350-450 laws).
2. **Spot-check 5 laws.** Pick the first 5 from the response and verify the fields: enacted date, law number, originating bill type + number. If the shape is consistent and `/law/119` returns the same shape, proceed. If shapes differ between Congresses, flag and stop.

Document the verified shape in the commit message.

## In scope

### Schema

One new table for the 118th data:

```sql
CREATE TABLE historical_laws (
  congress INTEGER NOT NULL,
  law_number TEXT NOT NULL,        -- e.g. "118-1" or however the API formats it
  source_bill_id TEXT,             -- matches bills.id format when applicable
  enacted_date TEXT NOT NULL,      -- ISO date
  title TEXT,
  PRIMARY KEY (congress, law_number)
);
CREATE INDEX idx_historical_laws_enacted ON historical_laws(congress, enacted_date);
```

Stores 118th only. The 119th's enacted laws stay in `bills` (where `stage='enacted'`); the chart query UNIONs the two sources. No duplicate storage for 119, no need to keep historical_laws in sync with bills.

### Backfill script

`scripts/backfill-historical-laws.ts`. One-time run. Paginates `/law/118`, upserts into `historical_laws`. Run via `npm run backfill:laws`.

If 118th turns out to have ~400 laws and ~2 pages of API response, total runtime should be under 30 seconds. No LLM calls; raw API data only.

### Query

`getLawsEnactedBySessionWeek()` in `lib/queries.ts`. Returns:

```ts
type LawsByWeek = {
  congress: 118 | 119;
  weekOfSession: number;     // 0 = first week after Jan 3 start
  cumulativeLaws: number;    // running total at end of week
}[];
```

The query UNIONs two CTEs:
- 118 from `historical_laws WHERE congress = 118`
- 119 from `bills WHERE congress = 119 AND stage = 'enacted'`

For each, computes `weekOfSession = (julianday(enacted_date) - julianday(congress_start_date)) / 7` and a running sum. Congress start dates: 118th = 2023-01-03, 119th = 2025-01-03.

If the CTE/window-function shape is awkward in libSQL, fall back to fetching the raw rows and computing the cumulative sum in TypeScript. Simpler, marginally less efficient — fine at this row count.

### Chart component

`components/LawsEnactedComparison.tsx`. Hand-rolled SVG (matches BillsTimeSeries / SponsorProductivityScatter per HO 99). Two cumulative lines, one per Congress.

Design:
- X-axis: weeks of session, 0 to ~104 (full 2-year Congress = 104 weeks)
- Y-axis: cumulative laws enacted
- 118th: full curve in `--text-muted` (it's history; less visual weight)
- 119th: partial curve up to current week in `--accent-amber-bright`
- A vertical "today" marker at the current 119th week, showing where the comparison sits
- Annotation at the marker: "119th: N laws · 118th at this point: M" or similar
- Light gridlines, axis labels in `--text-dim`, no chart-junk

The two existing SVG charts each re-declare their own axis/legend scaffolding. HO 99 flagged this for extraction "when the third SVG chart lands." This is that third chart. **In scope here**: pull the shared scaffolding into `components/svg/` primitives (`SvgAxis`, `SvgLegend`, or whatever lands cleanly), then use them for this chart and refactor BillsTimeSeries + SponsorProductivityScatter to consume them too. Don't ship a third one-off SVG.

If the refactor adds significant risk to BillsTimeSeries or SponsorProductivityScatter, ship the new chart with shared primitives and leave the existing two alone with a TODO. Don't break working charts to enforce DRY.

## Out of scope

- Backfilling 118th bills (full data, not just laws)
- Backfilling 117th or earlier Congresses
- Charting anything other than enacted laws (bills introduced, votes, etc. — separate charts, separate handoffs)
- Editorial overlays (party control differences, divided vs. unified government, etc.) — that's commentary, not a chart
- Wiring `/law/{congress}` into the cron — 118th is static history; 119th comes from the bills table which is already on cron via `/api/sync`

## Notes

- **Chart math edge case.** Some laws from the 118th have enacted dates in early January 2025 (lame-duck signings). The 118th formally ran Jan 3 2023 → Jan 3 2025. Anything enacted between Jan 3 2025 and current date belongs to the 119th. The simple `WHERE congress = 118` filter handles this correctly because the law's congress field is what it was assigned at enactment, not the calendar.
- **The `today` marker should match the 119th's current weekOfSession.** Compute it from `(today - 2025-01-03) / 7`. If today's date is in the 120th Congress (Jan 2027+), the chart needs a different framing — but that's 18+ months out; ignore for now.
